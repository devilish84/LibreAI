# LibreAI — Claude Code Guide

## Project Overview

LibreAI is a LibreOffice Writer extension providing an AI writing assistant. It is a **native C++17/Qt6 UNO component** (shared library) — no Java. It connects to Ollama (local), OpenAI, or Anthropic Claude and lets users chat, rewrite selected text, and apply AI responses back to the document.

**Current version:** 1.0.5  
**Branch strategy:** develop on `development`, release by tagging (`v*`) which triggers CI.

---

## Build & Install

```bash
bash build.sh            # configure + compile + package → libreai.oxt
bash build.sh --install  # also installs into LibreOffice via unopkg
```

Always kill LibreOffice before installing:
```bash
pkill -x soffice.bin; pkill -x soffice
```

**Build dependencies (Linux):**
```bash
sudo apt-get install build-essential cmake qt6-base-dev libreoffice-dev linguist-qt6
```

UNO headers must be pre-generated into `include/`:
```bash
cppumaker -O include \
    /usr/lib/libreoffice/program/types.rdb \
    /usr/lib/libreoffice/program/types/offapi.rdb
```

## Architecture

### Key Classes

| Class | File | Role |
|-------|------|------|
| `LibreAIJob` | `src/LibreAIJob.cpp` | UNO `XJobExecutor` — handles menu/toolbar/right-click triggers (`open`, `config`, `open_with_sel`) |
| `LibreAIStarter` | `src/LibreAIStarter.cpp` | UNO `XJob` — runs at LO startup; opens chat window, installs context-menu interceptor |
| `ChatWindow` | `src/ChatWindow.cpp` | Qt6 singleton chat window (`QWidget`, `Qt::Window`) |
| `ConfigDialog` | `src/ConfigDialog.cpp` | Qt6 singleton configuration dialog with two tabs |
| `CMInterceptor` | `src/CMInterceptor.cpp` | `XContextMenuInterceptor` — injects right-click item |
| `Config` | `src/Config.cpp` | JSON config singleton, saves to `~/.config/libreai/config.json` |
| `AIClient` | `src/AIClient.hpp` | Abstract base for provider clients (signals: `modelsReady`, `responseReady`, `errorOccurred`) |
| `OllamaClient` | `src/OllamaClient.cpp` | Ollama REST (`GET /api/tags`, `POST /api/chat`) |
| `OpenAIClient` | `src/OpenAIClient.cpp` | OpenAI REST (`GET /models`, `POST /chat/completions`) |
| `AnthropicClient` | `src/AnthropicClient.cpp` | Claude REST (`POST /v1/messages`; model list is hard-coded) |
| `UnoHelper` | `src/UnoHelper.cpp` | `getSelectedText()`, `applyText()`, `getCurrentFrame()` |

### Singleton Pattern

Both `ChatWindow` and `ConfigDialog` use a static member pointer (`s_instance`) with `instance()` and `resetInstance()`. `resetInstance()` closes and deletes the window; it is called when the user changes language so windows are recreated in the new language.

### Startup Sequence

```
LO starts
  → Jobs.xcu triggers LibreAIStarter on: onStartApp, onNew, onLoad, onViewCreated
      → LibreAIStarter::execute()
          → installInterceptors()    — iterate all frames, register CMInterceptor
          → attachDocumentListener() — once only; listens for OnViewCreated/OnNew/OnLoad
          → (first call only) create QApplication, apply language, show ChatWindow or ConfigDialog
```

`onViewCreated` is critical — it fires after the frame controller is fully ready, which is when `registerContextMenuInterceptor` actually succeeds.

### Interceptor De-duplication

`s_intercepted` (a `std::set<sal_IntPtr>`) tracks frame pointers. A frame is only added to the set **after** `registerContextMenuInterceptor` succeeds — if the controller isn't ready yet, the frame is not marked so it will be retried on the next event.

### Trigger Arguments

`LibreAIJob::trigger(args)`:
- `"open"` — check config, show ChatWindow (or ConfigDialog if unconfigured)
- `"config"` — show ConfigDialog
- `"open_with_sel"` — show ChatWindow with current Writer selection pre-loaded

## Configuration

Saved to `~/.config/libreai/config.json`:

```json
{
  "provider": "OLLAMA|OPENAI|CLAUDE",
  "ollama_url": "http://localhost:11434",
  "openai_url": "https://api.openai.com/v1",
  "openai_key": "",
  "claude_key": "",
  "model": "",
  "language": "en"
}
```

`Config::isConfigured()` returns true when a model is selected and the provider has the required credential. This gates opening ChatWindow — if false, ConfigDialog opens instead.

`Config::applyLanguage()` installs a `QTranslator` from the embedded `:/i18n/libreai_XX.qm` resource.

## Internationalization (i18n)

**Supported languages:** English (en), Finnish (fi), Swedish (sv), Danish (da), Norwegian (nb), Spanish (es), German (de), Portuguese (pt).

- Source files: `translations/libreai_XX.ts` — edit these to update translations
- Compiled files: `translations/libreai_XX.qm` — committed, regenerated by `lrelease` during build
- Embedded via: `translations/libreai_translations.qrc` → resource prefix `:/i18n/`
- Language selector: ConfigDialog → General Settings tab

**To add a new language:**
1. Copy `translations/libreai_fi.ts` → `translations/libreai_XX.ts`, translate all `<translation>` entries
2. Add `{"XX", "NativeName"}` to `kLanguages[]` in `src/ConfigDialog.cpp`
3. Add `<file>libreai_XX.qm</file>` to `translations/libreai_translations.qrc`
4. Add `<value xml:lang="XX">...</value>` for menu items in `Addons.xcu`
5. Run `lrelease translations/libreai_XX.ts` then `bash build.sh --install`

**Qt i18n conventions used:**
- `tr("...")` in `ChatWindow` and `ConfigDialog` (Q_OBJECT classes)
- `QCoreApplication::translate("CMInterceptor", "...")` in `CMInterceptor` (no Q_OBJECT)
- `retranslateUi()` + `changeEvent(QEvent::LanguageChange)` in both window classes for live retranslation

**LibreOffice menu items** (`Addons.xcu`) use native `xml:lang` attributes — they follow LibreOffice's own UI language, not LibreAI's language setting.

## Qt Inside LibreOffice — Important Constraints

- **`QApplication` is created once** in `LibreAIJob::trigger()` or `LibreAIStarter::execute()` if not already present. It is never destroyed.
- **All Qt windows use `Qt::Window` flag** — not `QDialog::exec()`, which would block LibreOffice's UNO thread and trigger the "not responding" watchdog.
- **All AI HTTP calls are async** via `QNetworkAccessManager` signals. Never use `QEventLoop` or blocking waits on the UNO thread.
- On **Windows**, Qt needs its plugin path set before `QApplication` creation — done via `GetModuleFileNameW(libreai_module_handle(), ...)` in `LibreAIJob.cpp`.
- `sal_Unicode*` vs `const ushort*`: on MSVC these are unrelated types. Always use `reinterpret_cast<const sal_Unicode*>(qstring.utf16())` when passing Qt strings to `rtl::OUString`.

## CI / Release

**Pipelines:**
- `.github/workflows/release.yml` — Linux (Ubuntu 22.04 + 24.04), triggered by `v*` tag push
- `.github/workflows/release-windows.yml` — Windows, triggered manually via `workflow_dispatch` with `tag` input

**Release workflow:**
```bash
git tag v1.0.X
git push origin development
git push origin v1.0.X                          # triggers Linux CI
gh workflow run release-windows.yml \
    --ref development --field tag=v1.0.X        # triggers Windows CI
```

If the tag needs to move (e.g. after a fix):
```bash
git push origin v1.0.X --delete
git tag -f v1.0.X
git push origin v1.0.X
gh workflow run release-windows.yml --ref development --field tag=v1.0.X
```

## Known Limitations

| Limitation | Detail |
|------------|--------|
| Anthropic model list | Hard-coded in `AnthropicClient.cpp` — Anthropic has no public list-models endpoint |
| Chat history | In-memory only; lost on LibreOffice restart |
| Writer only | `UnoHelper` uses `XTextViewCursorSupplier`, which is Writer-specific |
| Menu bar i18n | Follows LibreOffice's own UI language, not LibreAI's language setting |
| Qt event loop | Runs on LO's main thread; keep Qt operations lightweight |
